## 0. Research Focus

This document traces Trino 480's SIMD and vectorization strategy across the execution engine.
The central question: does Trino use explicit SIMD intrinsics, rely on JVM auto-vectorization,
or employ a hybrid? We trace the actual code paths to understand the real answer, and extract
the implications for a Rust-based replacement worker.

Key areas investigated:
- Explicit Java Vector API (`jdk.incubator.vector`) usage in block encoding serde and Parquet I/O
- SWAR (SIMD Within A Register) techniques in hash table probing (`FlatHash`)
- Bytecode-compiled columnar filter/projection loops and their auto-vectorization properties
- Null bitmap (MSB-first) handling and its vectorization story
- Hash computation (`XxHash64`) inner loops
- CPU feature detection and runtime SIMD capability switching

---

## 1. High-Level Overview

Trino 480 employs a **three-tier vectorization strategy**, not a single approach:

### Tier 1: Explicit Java Vector API (jdk.incubator.vector)
Used in two specific subsystems:
1. **Block encoding serde** (exchange data serialization/deserialization) -- uses `Vector.compress()`,
   `Vector.expand()`, `VectorMask`, and vectorized null-bit packing
2. **Parquet reader** -- uses `ByteVector`, `IntVector`, `ShortVector` for bit-unpacking,
   definition-level decoding, and boolean decoding

### Tier 2: SWAR (SIMD Within A Register)
Used in `FlatHash` for hash table probing -- a manual technique that packs 8 control bytes
into a single `long` and performs parallel byte matching using arithmetic tricks.

### Tier 3: JVM Auto-Vectorization (Implicit)
The bytecode-compiled filter and projection loops, type operators (comparison, hashing),
and most of the compute engine rely on HotSpot JIT to auto-vectorize tight loops.
Trino carefully structures these loops to be auto-vectorization-friendly (branchless updates,
contiguous memory access, fixed-width arrays).

### Mandatory Vector API Requirement
Trino 480 **requires** `jdk.incubator.vector` at startup -- `TrinoSystemRequirements.verifyVectorApiEnabled()`
checks `ModuleLayer.boot().findModule("jdk.incubator.vector")` and fails hard if absent. The JVM
flag `--add-modules=jdk.incubator.vector` is required in all configurations.

## 2. Detailed Analysis

### 2.1 Explicit SIMD: Block Encoding Serde (Exchange Path)

**File:** `core/trino-spi/src/main/java/io/trino/spi/block/EncoderUtil.java`
**File:** `core/trino-spi/src/main/java/io/trino/spi/block/{Byte,Short,Int,Long}ArrayBlockEncoding.java`
**File:** `core/trino-main/src/main/java/io/trino/simd/BlockEncodingSimdSupport.java`

When workers exchange data (shuffle), blocks must be serialized. Trino uses the Java Vector API
for three operations in this path:

#### 2.1.1 Null Bit Packing/Unpacking (EncoderUtil)

Trino stores nullability as a `boolean[]` in-memory but serializes it as MSB-first packed bits.
Two implementations exist side by side:

**Scalar path** (`encodeNullsAsBitsScalar`):
```java
// Manually packs 8 booleans into one byte with explicit shifts
int value = ((isNull[position + offset] ? 1 : 0) << 7) |
    ((isNull[position + offset + 1] ? 1 : 0) << 6) | ...
```

**Vectorized path** (`encodeNullsAsBitsVectorized`):
```java
// Uses ByteVector.SPECIES_64 (8 bytes = 8 booleans at once)
private static final ByteVector NULL_BIT_SHIFTS = ByteVector.fromArray(
    ByteVector.SPECIES_64, new byte[] {7, 6, 5, 4, 3, 2, 1, 0}, 0);

// Load 8 booleans, shift each to its bit position, OR-reduce to single byte
packedIsNull[currentByte++] = ByteVector.fromBooleanArray(ByteVector.SPECIES_64, isNull, position + offset)
    .lanewise(VectorOperators.LSHL, NULL_BIT_SHIFTS)
    .reduceLanes(VectorOperators.OR);
```

The decode direction uses the same `ByteVector.SPECIES_64` with right-shifts:
```java
ByteVector.broadcast(ByteVector.SPECIES_64, packedIsNull[currentByte++])
    .lanewise(VectorOperators.LSHR, NULL_BIT_SHIFTS)
    .intoBooleanArray(valueIsNull, position);
```

This is always enabled on x86_64 with >= 256-bit vector width (`packIsNullBits = true`).

#### 2.1.2 Null-Aware Value Compress/Expand

When serializing a block with nulls, non-null values must be compacted (compress) for
serialization and re-expanded (expand) on deserialization. This uses `Vector.compress(VectorMask)`
and `Vector.expand(VectorMask)` -- operations that map to hardware instructions:

- **x86 AVX-512F**: `VPCOMPRESSD` / `VPCOMPRESSQ` (int/long compress)
- **x86 AVX-512 VBMI2**: `VPCOMPRESSB` / `VPCOMPRESSW` (byte/short compress)
- **ARM SVE**: predicate-driven compress/expand

**IntArrayBlockEncoding compress example:**
```java
for (; valuesIndex < INT_SPECIES.loopBound(length); valuesIndex += INT_SPECIES.length()) {
    VectorMask<Integer> mask = INT_SPECIES.loadMask(isNull, valuesIndex + offset).not();
    IntVector.fromArray(INT_SPECIES, values, valuesIndex + offset)
        .compress(mask)
        .intoArray(compacted, compactedIndex);
    compactedIndex += mask.trueCount();
}
```

**IntArrayBlockEncoding expand example (branchless path):**
```java
for (; position < nonNullIndex && nonNullIndex + INT_SPECIES.length() < values.length;
     position += INT_SPECIES.length()) {
    IntVector nonNullValues = IntVector.fromArray(INT_SPECIES, values, nonNullIndex);
    VectorMask<Integer> nonNullMask = INT_SPECIES.loadMask(valueIsNull, position).not();
    nonNullIndex += nonNullMask.trueCount();
    nonNullValues.expand(nonNullMask).intoArray(values, position);
}
```

The expand path has two strategies for high-null-ratio data:
- **Selective loop** (when `nonNullCount * species.length * 4 <= positionCount`): adds a branch
  to skip all-null vectors entirely
- **Branchless loop** (default): always executes vector expand regardless of null content

#### 2.1.3 Runtime CPU Feature Detection

`BlockEncodingSimdSupport` performs runtime detection at startup via `MachineInfo.readCpuFlags()`:

**x86_64 detection:**
- Reads `/proc/cpuinfo` flags on Linux
- Checks for `avx512f` -> enables int/long compress+expand
- Checks for `avx512vbmi2` -> enables byte/short compress+expand
- Requires `preferredVectorBitWidth >= 256` (from `VectorShape.preferredShape()`)
- Null bit packing is always enabled on x86_64

**ARM detection:**
- Checks for `sve` flag (Graviton3+)
- NEON-only (Graviton2) gets `SimdSupport.NONE` -- emulated compress is too slow
- SVE enables compress for all types; expand only on `sve2`

**Wiring via Guice:**
`ServerMainModule` binds `BlockEncodingSimdSupport` as singleton. `BlockEncodingManager` reads
the detected `SimdSupport` record and passes per-capability booleans into each `BlockEncoding`
constructor. The config flag `exchange.experimental.vectorized-serde.enabled` (default: `true`)
can disable auto-detected support entirely.

### 2.2 Explicit SIMD: Parquet Reader

**File:** `lib/trino-parquet/src/main/java/io/trino/parquet/reader/decoders/VectorIntBitUnpackers.java`
**File:** `lib/trino-parquet/src/main/java/io/trino/parquet/reader/flat/VectorBitPackingUtils.java`
**File:** `lib/trino-parquet/src/main/java/io/trino/parquet/reader/flat/NullsDecoders.java`

Parquet uses bit-packing for dictionary IDs and definition/repetition levels. Trino provides
vectorized unpackers for bit widths 0-20 using the Java Vector API.

**Pattern (1-bit unpacker):**
```java
ByteVector.broadcast(ByteVector.SPECIES_64, input)     // replicate byte to 8 lanes
    .lanewise(VectorOperators.LSHR, LSHR_BYTE_VECTOR)  // shift each lane differently
    .and(MASK_1)                                        // mask to extract bit
    .castShape(IntVector.SPECIES_256, 0)                // widen byte -> int
    .reinterpretAsInts()
    .intoArray(output, outputOffset);                   // store 8 ints at once
```

Higher bit widths (3-bit example) use `rearrange()` (shuffle) + dual shift paths:
```java
ByteVector shiftRightResult = byteVector.rearrange(SHUFFLE_BEFORE_LSHR)
    .lanewise(VectorOperators.LSHR, LSHR_BYTE_VECTOR).and(MASK_3);
ByteVector shiftLeftResult = byteVector.rearrange(SHUFFLE_BEFORE_LSHL)
    .and(MASK_BEFORE_LSHL).lanewise(VectorOperators.LSHL, LSHL_BYTE_VECTOR);
shiftRightResult.or(shiftLeftResult)
    .castShape(IntVector.SPECIES_256, 0)
    .reinterpretAsInts().intoArray(output, outputOffset);
```

Each bit-width has its own dedicated `Unpacker` class with pre-computed constant vectors for
masks, shuffles, and shifts. The vectorized decoder is selected when `vectorizedDecodingEnabled`
is true and hardware has >= 256-bit vector width.

**Definition-level decoding** (`VectorNullsDecoder`) uses `vectorUnpackAndInvert8()`:
```java
ByteVector.broadcast(ByteVector.SPECIES_64, packedByte)
    .lanewise(VectorOperators.LSHR, LSHR_BYTE_VECTOR)
    .and(MASK_1)
    .lanewise(VectorOperators.NOT)
    .intoBooleanArray(values, offset);
```

### 2.3 SWAR: FlatHash Probing

**File:** `core/trino-main/src/main/java/io/trino/operator/FlatHash.java`

`FlatHash` is the Swiss-table-style hash map used by `FlatGroupByHash`, joins, and aggregations.
It uses a SWAR technique (not the Vector API) for control byte matching:

**Control byte layout:**
- Each hash table entry has a 1-byte control tag
- Empty slots = `0x00`, occupied slots = `(hash & 0x7F) | 0x80` (high bit always set)
- 8 control bytes are read as a single `long` via `VarHandle` with little-endian byte order

**`repeat(byte)` -- broadcast byte to all 8 positions in a long:**
```java
private static long repeat(byte value) {
    return ((value & 0xFF) * 0x01_01_01_01_01_01_01_01L);
}
```

**`match(long, long)` -- parallel 8-byte equality test (Hacker's Delight 6-1):**
```java
private static long match(long vector, long repeatedValue) {
    long comparison = vector ^ repeatedValue;
    return (comparison - 0x01_01_01_01_01_01_01_01L) & ~comparison & 0x80_80_80_80_80_80_80_80L;
}
```

This produces a bitmask with the high bit set in each byte that matched. The match result
is decoded via `Long.numberOfTrailingZeros(controlMatches) >>> 3` to find the matching slot
index within the 8-byte vector.

**Probing loop structure:**
```java
int step = 1;
long repeated = repeat(hashPrefix);
while (true) {
    long controlVector = (long) LONG_HANDLE.get(control, bucket);
    int matchIndex = matchInVector(blocks, position, hash, bucket, repeated, controlVector);
    if (matchIndex >= 0) return matchIndex;
    int emptyIndex = findEmptyInVector(controlVector, bucket);
    if (emptyIndex >= 0) return -emptyIndex - 1;
    bucket = bucket(bucket + step);
    step += VECTOR_LENGTH;  // triangular probing: step grows by 8 each iteration
}
```

The control array is allocated with `capacity + VECTOR_LENGTH` extra bytes to allow
over-reading at the end without bounds checks. Wrapping control bytes are duplicated
at the end: `control[index + capacity] = hashPrefix` for indices `< VECTOR_LENGTH`.

### 2.4 Bytecode-Compiled Filters and JIT Auto-Vectorization

**File:** `core/trino-main/src/main/java/io/trino/sql/gen/columnar/ColumnarFilterCompiler.java`
**File:** `core/trino-main/src/main/java/io/trino/sql/gen/columnar/CallColumnarFilterGenerator.java`
**File:** `core/trino-main/src/main/java/io/trino/sql/gen/columnar/IsNotNullColumnarFilter.java`

Trino's columnar filter path generates bytecode at query compile time. The generated code
produces tight loops that are candidates for HotSpot auto-vectorization:

**Generated filter range loop (non-null fast path):**
```java
// This is the bytecode pattern generated by CallColumnarFilterGenerator:
for (position = offset; position < offset + size; position++) {
    boolean result = call_function(position, block_0, block_1, ...);
    outputPositions[outputPositionsCount] = position;
    outputPositionsCount += result ? 1 : 0;   // branchless selection
}
```

Key auto-vectorization-friendly properties:
1. **Branchless output selection**: `outputPositionsCount += result ? 1 : 0` avoids a branch;
   the position is always written and the counter conditionally incremented. This maps to
   CMOV-style instructions that HotSpot JIT can vectorize.
2. **Null check hoisting**: The `mayHaveNull()` check is hoisted outside the loop. When no nulls
   exist, the inner loop has no null checks at all.
3. **Fixed-stride access**: Block data is stored in flat arrays (`long[]`, `int[]`, `byte[]`),
   giving the JIT contiguous memory access patterns.

However, several factors **limit** auto-vectorization of filter loops:
1. **Virtual dispatch**: `call_function` invokes via `invokedynamic` to a method handle; HotSpot
   must inline this before it can vectorize the surrounding loop.
2. **Non-trivial function bodies**: Comparison operators like `equal(long, long)` are simple
   enough to inline, but complex functions (regex, string operations) will prevent vectorization.
3. **Positions list mode**: When positions are gathered (not a contiguous range), the access
   pattern becomes `activePositions[index]` -> indirect/scatter, which prevents vectorization.

**IsNotNullColumnarFilter** is a hand-written filter (not bytecode-generated) that demonstrates
the ideal auto-vectorization pattern:
```java
byte[] isNull = isNullsBlock.get().getRawValues();
for (position = offset; position < offset + size; position++) {
    outputPositions[nonNullPositionsCount] = position;
    nonNullPositionsCount += isNull[isNullOffset + position] == 0 ? 1 : 0;
}
```
This is a textbook branchless selection loop over a contiguous `byte[]` array -- exactly the
pattern HotSpot can auto-vectorize.

### 2.5 Hash Computation (XxHash64)

**File:** `core/trino-spi/src/main/java/io/trino/spi/type/AbstractLongType.java`
**Dependency:** `io.airlift.slice.XxHash64` (external library)

Trino uses two hashing strategies:

**Fast hash (inline):**
```java
public static long hash(long value) {
    // xxHash64 mix
    return rotateLeft(value * 0xC2B2AE3D27D4EB4FL, 31) * 0x9E3779B185EBCA87L;
}
```
This is the `HASH_CODE` operator used for partitioning. It is a single multiply-rotate-multiply
sequence -- trivially auto-vectorizable by HotSpot when called in a loop.

**Full XxHash64 (external):**
```java
private static long xxHash64Operator(long value) {
    return XxHash64.hash(value);
}
```
Used for `XX_HASH_64` operator. The `XxHash64` class is in the `airlift.slice` library. For
single-value hashing, it is a scalar operation. For bulk operations (e.g., ORC `DictionaryBuilder`),
XxHash64 is called per-value in a loop -- the JVM may or may not vectorize depending on inlining
depth.

Neither hash function uses the Java Vector API. Hash computation in Trino is entirely scalar
or auto-vectorized.

### 2.6 Type Operators and Comparison Inner Loops

**File:** `core/trino-spi/src/main/java/io/trino/spi/type/AbstractLongType.java`
**File:** `core/trino-spi/src/main/java/io/trino/spi/type/TypeOperators.java`

Type operators (EQUAL, LESS_THAN, COMPARISON, XX_HASH_64) are defined as `@ScalarOperator`
annotated methods on type classes. They are resolved via `MethodHandle` and bound into generated
bytecode.

Example operators from `AbstractLongType`:
```java
@ScalarOperator(EQUAL)
private static boolean equalOperator(long left, long right) { return left == right; }

@ScalarOperator(LESS_THAN)
private static boolean lessThanOperator(long left, long right) { return left < right; }

@ScalarOperator(COMPARISON_UNORDERED_LAST)
private static long comparisonOperator(long left, long right) { return Long.compare(left, right); }
```

These are trivial scalar operations. When called from a tight filter loop over contiguous
`long[]` data, HotSpot can potentially auto-vectorize the entire loop including the operator.
However, the method handle indirection adds one layer that must be resolved by the JIT first.

### 2.7 Null Bitmap Impact on Vectorization

Trino's nullability is represented differently in different contexts:

**In-memory (blocks):** `boolean[] valueIsNull` -- one byte per position. This is NOT a bitmap;
it is a byte array where each element is `true` (1) or `false` (0). This representation is
vectorization-friendly for access patterns.

**On-wire (exchange serde):** MSB-first packed bits -- 8 positions per byte. The most significant
bit corresponds to the lowest position index within the group of 8.

**Impact on compute vectorization:**
The in-memory `boolean[]` representation means null checks in filter loops are simple byte
comparisons against zero, which auto-vectorize well:
```java
nonNullPositionsCount += isNull[isNullOffset + position] == 0 ? 1 : 0;
```

The packed bitmap representation only matters during serde, where Trino explicitly uses the
Vector API for packing/unpacking (Section 2.1.1). The MSB-first bit ordering is handled by
the pre-computed `NULL_BIT_SHIFTS = {7, 6, 5, 4, 3, 2, 1, 0}` constant vector.

The null bitmap does NOT prevent auto-vectorization of compute loops because blocks store
nulls as a flat `boolean[]`, not as packed bits.

## 3. Key Design Insights

### Insight 1: SIMD is Targeted, Not Pervasive
Trino does not vectorize the general compute engine with explicit SIMD. The Vector API is used
in exactly two subsystems: exchange serde (block encoding) and Parquet I/O. The core execution
engine (filters, projections, joins, aggregations) relies on loop structure and JIT.

### Insight 2: Dual-Path Architecture with Runtime Detection
Every SIMD-enabled path has a scalar fallback. The `BlockEncodingSimdSupport` record carries
per-operation capability flags (e.g., `compressByte`, `expandInt`). Each block encoding type
checks its specific flag and dispatches to the scalar or vectorized implementation. This allows
graceful degradation on hardware without AVX-512 or SVE.

### Insight 3: SWAR is Trino's Swiss-Table Implementation
`FlatHash` implements Google's Swiss-table algorithm in Java using SWAR rather than SIMD
intrinsics. The `repeat()`/`match()` functions perform 8-way parallel byte comparison in a
single `long`, achieving the same algorithmic effect as SSE2 `_mm_cmpeq_epi8`. This is Trino's
most performance-critical "vectorization" -- it runs on every hash probe in GROUP BY, JOIN,
and aggregation.

### Insight 4: Columnar Filter Loops are Auto-Vectorization-Optimized
The bytecode-generated filter loops use a consistent branchless selection pattern:
`output[count] = position; count += result ? 1 : 0`. This avoids branches in the inner loop,
enabling CMOV-style execution. The null check is hoisted out of the loop via `mayHaveNull()`.
These are deliberate design choices to help HotSpot's auto-vectorizer.

### Insight 5: Compress/Expand is the Key SIMD Operation
The dominant Vector API operation is `compress(mask)` / `expand(mask)`, which maps to
AVX-512 VPCOMPRESS/VPEXPAND instructions. This operation is uniquely useful for columnar data
with nulls -- it removes/reinserts null gaps in a single vector instruction. Without hardware
support, the JVM's emulation is slower than scalar code, which is why Trino explicitly detects
CPU flags before enabling it.

### Insight 6: Memory Layout Favors Vectorization
Trino stores fixed-width column data in flat Java arrays (`long[]`, `int[]`, `byte[]`), not
in off-heap buffers. This gives the JVM guaranteed contiguous memory and eliminates pointer
chasing. The `boolean[]` null representation (one byte per position, not packed bits) is
another deliberate choice that trades 8x memory for vectorization friendliness in compute.

## 4. Porting Considerations (Java -> Rust)

### 4.1 Replace SWAR with Native SIMD
The `FlatHash` SWAR technique is a workaround for Java's lack of stable SIMD intrinsics for
general compute. In Rust, replace this with:
- `std::arch::x86_64::_mm_cmpeq_epi8` (SSE2) or
- `std::simd::Simd<u8, 16>` (portable SIMD, nightly)

This gives 16-way parallel matching per instruction instead of 8-way, and avoids the arithmetic
overhead of the SWAR `repeat()`/`match()` functions. The Swiss-table crate (`hashbrown`) already
does exactly this.

### 4.2 Explicit SIMD for Block Serde
The compress/expand pattern maps directly to:
- **AVX-512**: `_mm512_mask_compressstoreu_epi32` / `_mm512_mask_expandloadu_epi32`
- **ARM SVE**: `svcompact` / predicated loads

In Rust, use `std::arch` intrinsics or the `packed_simd` / `std::simd` crate. Arrow's
`filter` kernel already implements this pattern for Arrow arrays.

### 4.3 Null Bitmap Representation Decision
Trino uses `boolean[]` (1 byte per position) in-memory for vectorization friendliness, but this
costs 8x memory vs packed bits. For the Rust worker:

**Option A: Apache Arrow validity bitmap** (packed bits, LSB-first)
- Pro: Memory efficient, industry standard, Arrow interop
- Con: Requires explicit SIMD or `pdep`/`pext` for masking in compute loops

**Option B: Byte-per-position** (Trino style)
- Pro: Trivially vectorizable, no bit manipulation overhead
- Con: 8x memory cost, non-standard

**Recommendation:** Use Arrow's packed bitmap and rely on Rust's auto-vectorization + explicit
SIMD for null handling. Rust's `u64` bitwise operations are efficient, and the Arrow ecosystem
provides optimized kernels. The 8x memory cost of byte-per-position is not justified when
explicit SIMD is available.

### 4.4 Filter Loop Vectorization Strategy
Trino's branchless `count += result ? 1 : 0` pattern translates naturally to Rust:
```rust
// Branchless selection (LLVM auto-vectorizes this)
output[count] = position;
count += filter_result as usize;
```

For even better performance, use explicit SIMD with mask-based scatter:
```rust
// With AVX-512 VPCOMPRESSD
let mask = simd_filter(block_data);
_mm512_mask_compressstoreu_epi32(output.as_mut_ptr(), mask, positions);
count += mask.count_ones();
```

### 4.5 Hash Computation
Trino's `XxHash64.hash(long)` is a single-value scalar hash. For the Rust worker:
- Use `xxhash-rust` crate for compatibility
- For bulk hashing (partition assignment), vectorize with explicit SIMD:
  - `ahash` provides AES-NI-based hashing that is faster than xxhash on modern CPUs
  - SIMD xxhash implementations exist that hash 4-8 values simultaneously

### 4.6 Parquet Bit-Unpacking
The `VectorIntBitUnpackers` pattern (broadcast + lane-specific shifts + mask) is a well-known
SIMD technique. In Rust:
- Use the `bitpacking` crate which already provides SIMD-optimized bit unpacking
- Or implement directly with `std::arch` using the same broadcast-shift-mask pattern
- Arrow Parquet reader already provides vectorized decoders

### 4.7 Summary: Recommended Rust SIMD Strategy

| Trino Component | Trino Approach | Rust Recommendation |
|---|---|---|
| Hash table probing | SWAR (manual 8-way) | `hashbrown` (SSE2/NEON 16-way) |
| Block serde compress/expand | Java Vector API | `std::arch` AVX-512/SVE intrinsics |
| Null bit packing | Java Vector API | Arrow bitmap utilities |
| Parquet bit unpacking | Java Vector API | `bitpacking` crate or Arrow Parquet |
| Filter/projection loops | JVM auto-vectorization | LLVM auto-vectorization + explicit SIMD kernels |
| Hash computation | Scalar XxHash64 | `xxhash-rust` + batch SIMD hashing |
| Type comparisons | Scalar + JIT inline | LLVM auto-vectorization (trivially vectorizable) |

**Overall recommendation:** Use a hybrid approach. Rely on LLVM auto-vectorization for
straightforward loops (comparisons, arithmetic), but use explicit SIMD for the critical paths
identified above (hash probing, null-aware compress/expand, bit unpacking). The Rust ecosystem
already provides high-quality SIMD implementations for most of these operations through
`hashbrown`, Arrow, and the `bitpacking` crate.
