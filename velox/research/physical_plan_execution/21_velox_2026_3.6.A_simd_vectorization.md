## 0. Research Focus

This document traces Velox's strategy for explicit SIMD (Single Instruction Multiple Data) intrinsics usage, focusing on:
- The **xsimd** abstraction library and how it provides cross-architecture portability
- SIMD-optimized hot paths: hash table probing, string operations, filter evaluation, bitmap manipulation, bloom filters
- The `SelectivityVector` bitmap-based row selection model that enables SIMD-friendly batch iteration
- The `StringView` inline-prefix layout that avoids pointer chasing for short-circuit comparisons
- Compile-time vs. runtime dispatch strategy
- Implications for porting these patterns to Rust

**Key source files examined:**
- `velox/common/base/SimdUtil.h` and `SimdUtil-inl.h` -- core SIMD abstraction layer
- `velox/common/base/BitUtil.h` -- word-at-a-time bitmap operations
- `velox/exec/HashTable.h` and `HashTable.cpp` -- SIMD tag-byte hash probing (the `ProbeState` class)
- `velox/type/Filter.h` -- SIMD-vectorized filter `testValues()` virtual methods
- `velox/type/StringView.h` -- inline-prefix string layout
- `velox/vector/SelectivityVector.h` -- bitmap-based row selection
- `velox/common/base/SplitBlockBloomFilter.h` -- SIMD bloom filter
- `velox/dwio/common/DecoderUtil.h` and `ColumnVisitors.h` -- SIMD filter pushdown into I/O decoding
- `velox/functions/lib/SIMDComparisonUtil.h` -- SIMD comparison vectorization

## 1. High-Level Overview

Velox's SIMD strategy has three layers:

```
+---------------------------------------------------------------+
|  Layer 3: Domain-Specific SIMD Hot Paths                      |
|  HashTable tag probing, Filter::testValues(), BloomFilter,    |
|  ColumnVisitors (I/O filter pushdown), simdStrstr, gather8Bits|
+---------------------------------------------------------------+
|  Layer 2: SimdUtil Abstraction Library                        |
|  gather/maskGather, filter/permute, toBitMask/fromBitMask,   |
|  memcpy, memset, leadingMask, transpose, iota, crc32U64,     |
|  memEqualUnsafe, simdFill, indicesOfSetBits                   |
+---------------------------------------------------------------+
|  Layer 1: xsimd 10.0.0 + Conditional Raw Intrinsics          |
|  xsimd::batch<T,A>, xsimd::batch_bool<T,A>,                  |
|  #if XSIMD_WITH_AVX2 / SSE2 / NEON / SVE compile guards      |
+---------------------------------------------------------------+
```

**Key architectural decisions:**

1. **xsimd as the primary abstraction** -- Velox uses xsimd 10.0.0 (from xtensor-stack) as its portable SIMD type system. `xsimd::batch<T>` and `xsimd::batch_bool<T>` provide load/store/broadcast/compare operations with a template parameter `A` for the architecture (defaulting to `xsimd::default_arch`).

2. **Compile-time dispatch, NOT runtime dispatch** -- There is no CPUID-based runtime dispatch. The architecture is selected at compile time via `xsimd::default_arch`. The `#if XSIMD_WITH_AVX2` / `XSIMD_WITH_SSE2` / `XSIMD_WITH_NEON` / `XSIMD_WITH_SVE` preprocessor guards select which specialization gets compiled. This means: one binary per target architecture.

3. **Raw intrinsics where xsimd falls short** -- For operations xsimd does not abstract well (gather, masked gather, movemask, permute, CRC32), Velox drops down to raw intrinsics (`_mm256_i32gather_epi32`, `_mm256_movemask_epi8`, `__builtin_crc32`, ARM inline asm) behind the compile-time guards, wrapped in SimdUtil's template specialization machinery.

4. **Targeted instruction sets:**
   - **x86-64**: SSE2, SSE4.1, SSE4.2, AVX, AVX2 (primary target, most optimized paths)
   - **ARM**: NEON (Apple Silicon / Graviton), SVE (128-bit and 256-bit variants)
   - **Generic fallback**: scalar loops for every operation, ensuring correctness on any platform
   - **Notably absent**: AVX-512 (Velox does not target it; sticks to 256-bit max)

## 2. Detailed Analysis

### 2.1 The SimdUtil Abstraction Layer

The `SimdUtil.h` / `SimdUtil-inl.h` pair (approximately 1950 lines) is the backbone. It provides a consistent C++ template API that dispatches to architecture-specific implementations via tag-dispatch on the `A` (architecture) type parameter.

**Core primitives provided:**

| Operation | Purpose | AVX2 Implementation | Fallback |
|-----------|---------|---------------------|----------|
| `toBitMask(batch_bool)` | Convert SIMD comparison result to scalar bitmask | `_mm256_movemask_epi8/ps/pd` | `genericToBitMask` (store + loop) |
| `fromBitMask(int)` | Convert scalar bitmask to `batch_bool` | Precomputed lookup table (`fromBitMask32[mask]`) | Same lookup table |
| `gather<T>(base, indices)` | Non-contiguous memory load | `_mm256_i32gather_epi32/64` | Scalar loop |
| `maskGather<T>(src, mask, base, indices)` | Conditional gather | `_mm256_mask_i32gather_epi32/64` | Scalar loop |
| `filter(data, bitMask)` | Compress/permute selected lanes to front | `_mm256_permutevar8x32_epi32` with lookup table | `genericPermute` |
| `leadingMask(n)` | First N lanes set | Precomputed lookup table (`leadingMask32[n]`) | Same |
| `crc32U64(checksum, value)` | Hardware CRC32 | `_mm_crc32_u64` (SSE4.2) | ARM: inline asm `crc32cx` |
| `memEqualUnsafe(x, y, size)` | SIMD memory equality | Batch compare + movemask + ctz | Same pattern |
| `simdStrstr(s, n, needle, k)` | SIMD substring search | AVX2 first+last char broadcast | Fallback to `std::find` |
| `memcpy(to, from, bytes)` | Wide-copy | Batch load/store at max width | Stepdown to 64/32/16/8 |
| `memset(to, data, bytes)` | Wide-fill | Broadcast + batch store | Stepdown |
| `simdFill(out, value, count)` | Typed array fill | Broadcast + store loop + overlapping tail | `std::fill` |
| `indicesOfSetBits(bits, begin, end, indices)` | Bitmap -> position list | Load `byteSetBits[byte]` as batch, add row offset, store | Scalar ctz loop |
| `transpose(input, indices, output)` | Gather-based permutation | Batch gather + store | Scalar loop |
| `gather8Bits(bits, indices, n)` | Gather disjoint bits from bitmap | Gather bytes, permute bit masks, AND, compare | Scalar `isBitSet` loop |
| `storeLeading(data, mask, n, dest)` | Masked store of leading N lanes | `_mm256_maskstore_epi32/64` | Scalar loop |

**The precomputed lookup table pattern** is important: `byteSetBits[256][8]` stores, for every possible byte value, the positions of set bits. This enables O(1) conversion from a byte bitmask to a permutation index vector, which is critical for the `filter` (compress) operation that rearranges passing values to the front of a vector.

**Padding convention**: `simd::kPadding = max(32, batchByteSize())` ensures buffers always have enough trailing space for full-width SIMD writes past logical boundaries. This avoids boundary checks in hot loops.

### 2.2 SIMD Hash Table Probing (The Crown Jewel)

The hash table in `HashTable.h`/`HashTable.cpp` uses a **Swiss-table-inspired** design with 16-byte tag vectors for SIMD probing. This is Velox's most performance-critical SIMD path.

**Bucket Layout (128 bytes = 2 cache lines):**

```
struct Bucket {            // 128 bytes total
    TagVector tags_;       // 16 x uint8_t = 16 bytes (SSE2 register / NEON register)
    char pointers_[96];    // 16 x 6-byte truncated pointers
    char padding_[16];     // alignment padding
};
```

Key design: `TagVector` is always 128-bit (SSE2 on x86, NEON on ARM), regardless of the wider SIMD support. This is because the bucket contains exactly 16 slots, and 16 x 1-byte tags fit perfectly in a 128-bit register.

**Tag extraction**: `hashTag(hash)` extracts bits 38-44 from the hash and sets the high bit:
```cpp
static uint8_t hashTag(uint64_t hash) {
    return static_cast<uint8_t>(hash >> 38) | 0x80;
}
```
The `| 0x80` guarantees the tag is never 0 (empty) or 0x7f (tombstone), so tag == 0 reliably means "empty slot".

**The ProbeState SIMD probe sequence:**

```
Step 1: preProbe() -- broadcast the search tag to all 16 lanes
    wantedTags_ = TagVector::broadcast(tag);  // 1 instruction
    __builtin_prefetch(table + bucketOffset);  // prefetch the bucket

Step 2: firstProbe() -- load 16 tags and compare in 2 instructions
    tagsInTable_ = loadTags(table, offset);   // 1 load (SSE2 movdqu / NEON vld1q_u8)
    hits_ = toBitMask(tagsInTable_ == wantedTags_);  // 1 compare + 1 movemask

Step 3: fullProbe() -- iterate matching slots, check for empties/tombstones
    while (hits_ > 0) {
        loadNextHit(table, firstKey);  // getAndClearLastSetBit, load pointer
        if (compare(group_, row_))     // full key comparison
            return group_;
    }
    // Check for empty slots (probe termination)
    uint16_t empty = toBitMask(tagsInTable_ == kEmptyGroup);
    if (empty > 0) return nullptr;     // not found
    // Move to next bucket (linear probing)
```

**Four-way interleaved probing**: The `groupProbe()` function processes 4 rows simultaneously, interleaving their preProbe/firstProbe/fullProbe stages to hide memory latency:

```cpp
for (; probeIndex + 4 <= numProbes; probeIndex += 4) {
    state1.preProbe(*this, hashes[rows[probeIndex]], rows[probeIndex]);
    state2.preProbe(*this, hashes[rows[probeIndex+1]], rows[probeIndex+1]);
    state3.preProbe(*this, hashes[rows[probeIndex+2]], rows[probeIndex+2]);
    state4.preProbe(*this, hashes[rows[probeIndex+3]], rows[probeIndex+3]);
    // By the time we reach firstProbe for state1, its bucket is likely in cache
    state1.firstProbe(*this, 0);
    state2.firstProbe(*this, 0);
    state3.firstProbe(*this, 0);
    state4.firstProbe(*this, 0);
    // ...
}
```

This software pipelining means 4 prefetches are issued before any probe reads the data, giving the memory subsystem time to fetch all 4 buckets.

**NormalizedKey fast path**: For simple key types, Velox stores a concatenated 64-bit "normalized key" before each row. The `joinNormalizedKeyFullProbe()` compares normalized keys directly without accessing the full row, avoiding one level of indirection:
```cpp
if (RowContainer::normalizedKey(group_) == keys[row_]) {
    return group_;
}
```

### 2.3 SIMD Filter Evaluation

Velox's `Filter` class hierarchy in `Filter.h` defines virtual `testValues(xsimd::batch<T>)` methods that apply a filter predicate to an entire SIMD vector at once.

**BigintRange filter (range check):**
```cpp
xsimd::batch_bool<int64_t> testValues(xsimd::batch<int64_t> values) const final {
    if (singleValue_) {
        return values == xsimd::broadcast<int64_t>(lower_);
    }
    return (xsimd::broadcast<int64_t>(lower_) <= values) &
           (values <= xsimd::broadcast<int64_t>(upper_));
}
```

Two SIMD comparisons + one AND produce the filter result for 4 int64 values (AVX2) or 2 (SSE2) simultaneously.

**BigintValuesUsingHashTable filter (IN-list via hash probing):**
This is the most sophisticated SIMD filter. For `WHERE x IN (v1, v2, ..., vN)`, Velox builds a hash table of the IN-list values, then probes it with SIMD gather:

```cpp
xsimd::batch_bool<int64_t> testValues(xsimd::batch<int64_t> x) const final {
    // Step 1: Range check (SIMD broadcast compare)
    auto outOfRange = (x < broadcast(min_)) | (x > broadcast(max_));
    if (toBitMask(outOfRange) == allSetBitMask<int64_t>())
        return batch_bool(false);  // all out of range

    // Step 2: Hash probe via SIMD gather
    auto indices = reinterpretBatch<int64_t>(reinterpretBatch<uint64_t>(x) * M & sizeMask_);
    auto data = maskGather(allEmpty, ~outOfRange, hashTable_.data(), indices);

    // Step 3: Compare gathered values against input
    auto result = x == data;
    auto missed = data == allEmpty;

    // Step 4: Handle collisions (scalar fallback per unresolved lane)
    uint16_t unresolved = allSetBitMask ^ (toBitMask(result) | toBitMask(missed));
    while (unresolved) {
        auto lane = getAndClearLastSetBit(unresolved);
        // Scalar linear probe for this lane...
    }
    return fromBitMask<int64_t>(resultBits);
}
```

This is a vectorized hash probe: 4 hash table lookups happen in parallel via `maskGather`, and only collision cases fall back to scalar.

**processFixedFilter (I/O layer):**
The `DecoderUtil.h` `processFixedFilter` function is the main SIMD filter path used during column reading. It:
1. Calls `filter.testValues(values)` to get a SIMD boolean mask
2. Converts to a bitmask with `toBitMask()`
3. Uses `simd::filter()` to compress passing row indices and values to the front
4. Stores them contiguously for the next pipeline stage

```cpp
auto word = simd::toBitMask(filter.testValues(values));
if (!word) { /* no values passed */ }
else if (word == simd::allSetBitMask<T>()) {
    // All passed -- direct store, no shuffling
    loadIndices(0).store_unaligned(filterHits + numValues);
    values.store_unaligned(rawValues + numValues);
    numValues += width;
} else {
    // Partial pass -- SIMD permute to compress
    simd::filter<int32_t>(loadIndices(0), bits).store_unaligned(filterHits + numValues);
    simd::filter(values, allBits).store_unaligned(rawValues + numValues);
    numValues += __builtin_popcount(allBits);
}
```

### 2.4 SelectivityVector: Bitmap-Based SIMD-Friendly Row Selection

`SelectivityVector` is NOT itself a SIMD type -- it is a `std::vector<uint64_t>` bitmap with `begin_`/`end_` bounds tracking. Its SIMD-friendliness comes from how it interacts with processing loops.

**Key design elements:**

1. **`allSelected_` fast path**: When all rows are selected (common case), `applyToSelected()` uses a simple sequential loop that the compiler can auto-vectorize:
```cpp
if (isAllSelected()) {
    const auto end = end_;  // IMPORTANT: helps compiler vectorize
    for (vector_size_t row = begin_; row < end; ++row) {
        func(row);
    }
}
```

2. **Sparse iteration via `forEachSetBit`**: When rows are sparse, `bits::forEachSetBit` uses `__builtin_ctzll` (count trailing zeros) to jump directly to set bits, skipping zero words entirely:
```cpp
while (current_ == 0) {
    current_ = bits_[++index_];
}
row = (index_ * 64) + __builtin_ctzll(current_);
current_ &= current_ - 1;  // clear lowest set bit
```

3. **`forBatches<kWidth>` for SIMD-width iteration**: `BitUtil.h` provides `forBatches<8>` that iterates over the bitmap in groups of 8 (or whatever SIMD width), calling a function with the base index and a sub-mask of active rows. This enables SIMD operations to process only the active portions of the bitmap.

4. **Bulk boolean operations**: `intersect()`, `deselect()`, `select()`, `deselectNulls()` use word-at-a-time `andBits`/`orBits`/`andWithNegatedBits` operations on the underlying `uint64_t` arrays, which process 64 rows per instruction.

### 2.5 StringView Prefix Layout for SIMD-Friendly Comparison

Velox's `StringView` (16 bytes total) uses an inline-prefix design:

```
+--------+--------+-------------------+
| size   | prefix | pointer/inline    |
| 4 bytes| 4 bytes| 8 bytes           |
+--------+--------+-------------------+

Strings <= 12 bytes: fully inlined (prefix + inline data)
Strings > 12 bytes: prefix has first 4 chars, pointer to full data
```

**SIMD-friendly comparison via `sizeAndPrefixAsInt64()`:**
```cpp
bool operator==(const StringView& other) const {
    // Single 64-bit compare checks both size AND first 4 chars
    if (sizeAndPrefixAsInt64() != other.sizeAndPrefixAsInt64()) {
        return false;  // fast reject: ~90% of inequalities caught here
    }
    if (isInline()) {
        // Second 64-bit compare for the remaining inline data
        return size_ <= kPrefixSize || inlinedAsInt64() == other.inlinedAsInt64();
    }
    // Out-of-line: memcmp the remainder past the 4-byte prefix
    return memcmp(value_.data + kPrefixSize, other.value_.data + kPrefixSize,
                  size_ - kPrefixSize) == 0;
}
```

This design means:
- **Equality of short strings (<=12 bytes)** requires at most 2 integer comparisons (no pointer chasing)
- **Inequality** is usually detected in the first 8-byte compare (size + prefix mismatch)
- **Sorting/comparison** uses `prefixAsInt()` for the first branch, resolving most cases without touching the heap
- The inline data is **zeroed at construction**, enabling whole-word comparison without worrying about padding bytes

### 2.6 SIMD Bloom Filter

`SplitBlockBloomFilter` is a fully SIMD-native data structure. Each block is a SIMD register (`xsimd::batch<uint32_t>` = 8 lanes on AVX2, 4 lanes on SSE2/NEON).

**Insert:**
```cpp
void insert(uint64_t hash) {
    auto mask = makeMask(hash);
    auto* block = blocks_[blockIndex(hash)].data;
    (xsimd::load_aligned(block) | mask).store_aligned(block);
}
```

**Probe:**
```cpp
bool mayContain(uint64_t hash) const {
    auto mask = makeMask(hash);
    auto block = xsimd::load_aligned(blocks_[blockIndex(hash)].data);
    return _mm256_testc_si256(block, mask);  // AVX: all mask bits present?
}
```

The `makeMask` function uses SIMD multiply + shift to derive 8 independent bit positions from the hash, then sets those bits. Probing checks all 8 bits in a single `testc` instruction on AVX.

### 2.7 SIMD String Search (simdStrstr)

Velox implements a SIMD-accelerated substring search that is ~2x faster than `std::string_view::find()`:

```cpp
// Broadcast first and last character of needle
auto first = CharVector::broadcast(needle[0]);
auto last = CharVector::broadcast(needle[needleSize - 1]);

for (i = 0; i <= n - needleSize - CharVector::size; i += CharVector::size) {
    auto blockFirst = CharVector::load_unaligned(s + i);
    auto eqFirst = (first == blockFirst);    // 32 char comparisons at once (AVX2)
    if (eqFirst.mask() == 0) continue;       // no first-char match: skip 32 chars

    auto blockLast = CharVector::load_unaligned(s + i + needleSize - 1);
    auto eqLast = (last == blockLast);
    auto mask = (eqFirst && eqLast).mask();  // positions where first AND last match

    while (mask != 0) {
        auto bitpos = __builtin_ctz(mask);
        if (memcmp(s + i + bitpos + 1, needle + 1, needleSize - 2) == 0)
            return i + bitpos;
        mask &= mask - 1;
    }
}
```

The algorithm uses compile-time template specialization for needle sizes 2-15 (via a `switch` + macro `VELOX_SIMD_STRSTR_CASE`) to enable compile-time constant propagation in the `memcmp` call.

### 2.8 CRC32 Hardware Acceleration

Velox uses hardware CRC32C for hashing via `simd::crc32U64()`:
- **x86 (SSE4.2)**: `_mm_crc32_u64` intrinsic
- **ARM (NEON/SVE)**: Inline assembly `crc32cx` instruction
- No software fallback is provided -- CRC32 is expected to be available on all target platforms

`VectorHasher` uses this for string hashing with a 3-way unrolled pipeline:
```cpp
a0 = simd::crc32U64(a0, words[0]);
a1 = simd::crc32U64(a1, words[1]);
a2 = simd::crc32U64(a2, words[2]);
```
The 3-way unrolling hides the latency of the CRC instruction (3 cycles on modern x86) by keeping 3 independent hash chains in flight.

### 2.9 SIMD Memory Operations

Velox provides SIMD-optimized `memcpy`, `memset`, and `simdFill` that operate at full SIMD width:

- `simd::memcpy`: Copies in batch-width chunks (32 bytes on AVX2), stepping down through 8/4/2/1 byte copies for the tail
- `simd::memset`: Broadcasts the fill byte and stores in batch-width chunks
- `simd::simdFill<T>`: Broadcasts a typed value and stores in batch-width chunks with an overlapping tail store (avoids branch for non-aligned remainder)
- `simd::memEqualUnsafe`: Compares memory in batch-width chunks using `load_unaligned + compare + movemask`, stopping at the first mismatch

### 2.10 Null Bitmap Gathering (gather8Bits)

The `gather8Bits` function is a specialized operation for reading null bits at non-contiguous positions -- a common operation when evaluating nullability of filtered/scattered rows.

On AVX2, it uses a single SIMD gather + bit manipulation:
```cpp
// Load the byte containing each bit, extract the specific bit
auto maskV = Permute(kByteBits, vindex & 7);      // bit mask per lane
auto data = maskGather(zero, mask, bits, vindex >> 3);  // gather bytes
return allSetBitMask ^ toBitMask((data & maskV) == zero);  // test bits
```

This replaces 8 individual `isBitSet` calls with a single gather + AND + compare sequence.

## 3. Key Design Insights

1. **Compile-time dispatch is the intentional choice.** Velox does not use runtime CPUID dispatch. The template parameter `A = xsimd::default_arch` is resolved at compile time based on compiler flags. This trades binary portability for zero-overhead dispatch. Multiple binaries may be compiled for different targets, but there is no runtime function pointer indirection.

2. **AVX-512 is intentionally avoided.** Despite its wider registers, AVX-512 causes frequency throttling on many Intel CPUs and has inconsistent support. Velox caps at 256-bit (AVX2) as the optimal performance/compatibility tradeoff. The hash table tag comparison explicitly uses 128-bit (16 byte = 16 tags per bucket), fitting exactly one SSE2/NEON register.

3. **Precomputed lookup tables are pervasive.** `byteSetBits[256][8]`, `fromBitMask32/64`, `leadingMask32/64`, and `permute4x64Indices` are all precomputed at static initialization. This turns variable-dependent operations (e.g., "which bits are set in this mask?") into constant-time table lookups, which is critical when these operations occur inside tight inner loops.

4. **The "generic fallback" guarantee.** Every SIMD path has a `xsimd::generic` fallback using scalar loops. This means the code compiles and runs correctly on any platform, even without SIMD support. The specializations are purely performance optimizations layered on top.

5. **The four-way interleaved probe pattern is a software pipeline.** By issuing 4 prefetches, then 4 tag-loads, then 4 full-probes, Velox hides memory latency. The first bucket is in L1 by the time its full probe executes. This pattern is as important as the SIMD tag comparison itself.

6. **StringView's 8-byte `sizeAndPrefix` is a deliberate SIMD-adjacent optimization.** While not a SIMD operation per se, packing size + 4-byte prefix into a single 64-bit compare enables the CPU's scalar integer pipeline to reject ~90% of string inequalities in a single cycle. For the remaining cases where the prefix matches, the inlined continuation avoids a pointer chase for strings up to 12 bytes.

7. **The filter compress (simd::filter) operation is the key to vectorized selection.** Converting a comparison bitmask to a contiguous array of passing values/indices via permute is used everywhere: filter pushdown in I/O, hash table insert, comparison operators. The 256-entry permute lookup table (`byteSetBits`) makes this O(1) per byte of mask.

8. **SIMD is used in I/O decoding, not just execution.** `processFixedFilter` in `DecoderUtil.h` and the `ColumnVisitors.h` framework apply SIMD filters during Parquet/ORC decoding, before values even reach the execution engine. This is a tight integration where SIMD accelerates the entire data path from disk to result.

## 4. Porting Considerations (Velox C++ -> Trino Rust Worker)

### 4.1 SIMD Library Choice for Rust

| Velox Approach | Rust Equivalent | Notes |
|----------------|-----------------|-------|
| xsimd 10.0.0 (C++ templates) | `std::simd` (nightly) or `packed_simd2` | `std::simd` is the long-term target; nightly-only as of 2026 |
| `#if XSIMD_WITH_AVX2` compile guards | `#[cfg(target_feature = "avx2")]` | Rust has native target-feature compile-time selection |
| Raw `_mm256_*` intrinsics | `std::arch::x86_64::_mm256_*` | Available in stable Rust via `core::arch` |
| `xsimd::default_arch` template parameter | `#[target_feature(enable = "avx2")]` on functions | Rust uses function-level feature gates |

**Recommendation**: Use `std::arch` intrinsics wrapped in a Rust abstraction layer analogous to `SimdUtil`. The abstraction should use `#[cfg(target_feature)]` for compile-time dispatch. `std::simd` (portable SIMD) is ideal but may not be stable by the time of implementation -- plan for `std::arch` as the baseline with `std::simd` as a future migration path.

### 4.2 Hash Table Tag Probing

The Swiss-table tag probing pattern maps directly to Rust:
- `hashbrown` (the crate underlying `std::collections::HashMap`) already uses this exact SIMD probing pattern with SSE2/NEON tag comparison
- For a custom hash table, the pattern is: `u8x16::load(tags) == u8x16::splat(target)` -> `movemask` -> `ctz` bit scan
- The 4-way interleaved probing is a manual optimization that Rust can replicate with explicit prefetch (`core::arch::x86_64::_mm_prefetch`)

### 4.3 SelectivityVector -> Rust Bitmap

Rust already has excellent bitmap crates (`bitvec`, `roaring`), but the key insight from Velox is:
- Use `u64` words directly (not bit-level abstractions) for bulk operations
- Track `begin`/`end` bounds to skip empty ranges
- The `allSelected` fast path (sequential loop vs. sparse iteration) is critical for auto-vectorization
- `__builtin_ctzll` maps to `u64::trailing_zeros()` in Rust

### 4.4 StringView Layout

Velox's `StringView` layout (size + 4-byte prefix + 8-byte pointer/inline) can be replicated exactly in Rust:
```rust
#[repr(C)]
struct StringView {
    size: u32,
    prefix: [u8; 4],
    data: StringData,  // union: inline [u8; 8] or *const u8
}
```
The `sizeAndPrefixAsInt64()` trick maps to `unsafe { *(self as *const _ as *const u64) }`.

### 4.5 Key Porting Risks

1. **No runtime dispatch needed initially.** Like Velox, the Rust worker can target a single ISA (AVX2 for x86-64, NEON for ARM). Runtime dispatch adds complexity with minimal benefit for server-side workloads where the target CPU is known at deployment.

2. **Gather instructions are the main gap.** Rust's `std::simd` does not yet have portable gather. Use `_mm256_i32gather_epi32` via `core::arch` for the hash table IN-list filter and null-bit gathering paths.

3. **The lookup table pattern is zero-cost in Rust.** Static `[u8; 256]` arrays and `const fn` initialization replicate the `byteSetBits` tables. Rust's guaranteed zero-cost abstractions make this straightforward.

4. **`unsafe` will be required for SIMD hot paths.** The intrinsic functions in `core::arch` are all `unsafe`. Encapsulate them in a safe wrapper module (analogous to `SimdUtil`) with `debug_assert!` bounds checks.

5. **Auto-vectorization is not a substitute.** Velox's explicit SIMD code exists precisely because auto-vectorization is unreliable for complex patterns (gather, filter/compress, interleaved probing). The Rust worker should likewise use explicit intrinsics for the hot paths identified here, and rely on auto-vectorization only for simple sequential loops.
